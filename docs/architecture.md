# Runtime Architecture

## Main Rule

Audio leads. Audio generation must never wait on UI, logs, keyboard, SD,
graphics, or long locks.

## Proposed FreeRTOS Tasks

| Task | Priority | Responsibility |
| --- | --- | --- |
| `AudioTask` | High | Render buffers and write to I2S/speaker. |
| `InputTask` | Medium | Scan the keyboard and emit change events. |
| `SynthControlTask` | Medium | Apply note on/off, waveform, volume, and state changes. |
| `UiTask` | Low | Redraw visible state only when it changes. |
| `ChordTask` | Low/medium | Recalculate chords when active notes change. |

`ChordTask` can initially be integrated into `SynthControlTask`, which is how
the current firmware is organized.

## Module Boundaries

- `synth_engine` applies events and owns note lifecycle changes.
- `synth_envelope` owns the per-note ADSR envelope state machine.
- `synth_note_policy` owns small musical policies such as note identity,
  velocity gain, sustain polarity, and pitch bend scaling.
- `audio_render` renders a bounded snapshot and does not log, allocate, touch
  WiFi/USB, or draw.
- `dev_mode` owns HTTP/WiFi/OTA development infrastructure.
- `usb_midi_host` owns USB Host enumeration, USB MIDI packets, and optional M32
  OLED transport.

## Data Flow

```text
Cardputer keyboard
  -> InputTask
  -> SynthEvent queue
  -> SynthControlTask
  -> small copyable SynthAudioState
  -> AudioTask render
  -> I2S / speaker

SynthControlTask
  -> UiState
  -> UiTask
  -> Cardputer display
```

## Input Events

```cpp
enum class SynthEventType {
  NoteOn,
  NoteOff,
  SetWaveform,
  AdjustVolume,
  AdjustAttack,
  AdjustDecay,
  AdjustSustain,
  AdjustRelease,
  ControlChange,
  PitchBend
};

struct SynthEvent {
  SynthEventType type;
  uint8_t noteIndex;
  uint8_t midi;
  uint8_t velocity;
  Waveform waveform;
  float value;
  uint8_t control;
  uint8_t controlValue;
  int16_t pitchBend;
};
```

`InputTask` must send events without blocking:

```cpp
xQueueSend(synthEventQueue, &event, 0);
```

## Synth State

```cpp
enum class Waveform {
  Sine,
  Square,
  Rectangle,
  Saw
};

struct ActiveNote {
  bool active;
  uint8_t noteIndex;
  uint8_t midi;
  uint8_t velocity;
  float velocityGain;
  float frequency;
  float phase;
  float phaseIncrement;
  EnvelopeState envelope;
  uint32_t ageSamples;
  bool keyReleased;
};

struct SynthAudioState {
  Waveform waveform;
  float masterVolume;
  uint8_t activeCount;
  uint32_t pressedMask;
  bool sustainPedal;
  int16_t pitchBendRaw;
  float pitchBendMultiplier;
  EnvelopeParams ampEnvelope;
  uint8_t cc[128];
  ActiveNote notes[MAX_POLYPHONY];
};
```

Iteration 1 uses a small state copy protected by short critical sections. If
glitches appear, move to double-buffered state:

```text
controlState -> short atomic copy -> audioState
```

## AudioTask Rules

The audio path must not contain:

- `printf` or `ESP_LOG*`.
- `malloc`, `new`, dynamic `std::vector`, or `String`.
- SD reads.
- Display redraws.
- Long mutex waits.
- Slow I2C operations.

Expected structure:

```cpp
while (true) {
  copyAudioState(&localState);
  renderAudioBuffer(&localState, buffer, AUDIO_BUFFER_FRAMES);
  storeRenderedAudioState(localState);
  writeAudioFrames(buffer, AUDIO_BUFFER_FRAMES);
}
```

## Render And Output

- Internal format: normalized `float` in `[-1.0, 1.0]`.
- Final output: `int16`.
- Initial sample rate: 22050 Hz.
- Initial buffer: 128 frames.
- Maximum polyphony: 8 notes.
- Audio render applies fixed waveform loudness trims and a per-note ADSR
  amplitude envelope before velocity gain and mixing.

Conceptual conversion:

```cpp
int16_t toInt16(float sample) {
  if (sample > 1.0f) sample = 1.0f;
  if (sample < -1.0f) sample = -1.0f;
  return (int16_t)(sample * 32767.0f);
}
```

## Polyphony Policy

When more than 8 notes are requested, the engine uses a bounded voice
assignment policy:

1. Use an inactive slot.
2. Reuse a released duplicate note for retriggering.
3. Steal a voice in Release, preferring the quietest one.
4. If no Release voice exists, steal the oldest active voice.

Released notes continue to count as active until their envelope reaches Off.

## UI And Visual State

The UI must never derive pressed notes from transient events only. It must read
stable state generated by control:

- Active notes.
- Active waveform.
- Master volume.
- `n/8` count.
- Chord name.

Target UI frequency:

- 15-20 FPS maximum.
- Redraw only when important state changes.
